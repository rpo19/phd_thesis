We also borrow the \textbf{availability} requirement from \textcite{BELLANDI2024}, which aims at ensuring that the \ac{dia} does not suffer from changes or failures in other services such as extractors---i.e., \acl{ee} tools that annotate documents with entities---or \aclp{ui}. \begin{itemize}

    \item \textbf{Interoperability}, to support complex workflows in which several extractors are executed in sequence. For this reason, it is necessary that all services \emph{communicate} using a \ac{uniof}, so that sequential workflows can be seamlessly modified by adding or removing extractors.
\end{itemize}

\paragraph{Document--Annotation--Entity Triad}
Before introducing the functional requirements that support domain-specific use cases, we first define the three conceptual pillars that underpin our representation of heterogeneous textual data: \emph{documents}, \emph{annotations}, and \emph{entities}.

A \emph{document} is the abstract unit of textual data in our model. Its granularity is not fixed and may range from entire legal judgments to individual chat messages, depending on the use case. Documents serve as the primary carriers of text to which structured information can be attached.

An \emph{entity}, defined in \Cref{back:entities}, represents conceptual objects of interest that can be mentioned across documents. Entities enable the integration of heterogeneous sources into a unified knowledge representation, supporting the generalizability requirement.

An \emph{annotation} establishes a structured link between a document and a portion of its content, optionally associating it with an entity. Annotations serve as the mechanism that enables traceability from structured representations back to their textual evidence (traceability and verifiability), and they generalize beyond entity mentions to any meaningful fragment of text. For example they can represent document sections, or events.

We now introduce the additional functional requirements that support the domain-specific use cases considered in this thesis. As anticipated in \Cref{sec:intro}, these use cases include: (i) case retrieval and precedent search, (ii) document navigation and content exploration, (iii) investigative retrieval, (iv) question answering over documents and collections, and (v) statistical analysis and performance monitoring. The additional functional requirements are the following:
\begin{enumerate}
    \item \emph{\acf{crud}~\cite{martin1983managing} \acp{api}} for creating, reading, updating, deleting documents, entities, and annotations. However, an important exceptions need to be applied: since the annotations refer and depend on the document text, modifying it should normally be forbidden as it invalidates all the existing annotations. \todo{what happens if i modify a document with annotations?} % uri already required by the data model
    \item Support for \emph{multiple collections of documents}. While, for example, storing data from different investigations in the same \ac{dia} instace may not be advisable due to risks in data protection, this requirement allows to handle different---and interlink via \acp{uri}---different but related set of documents, or to host multiple \say{models} of the same data. As a pratical exampled derived from the investigative consultancy of \textcite{pozzi_wise_2025}, investigator may need to visualize entire chats from \ac{ima} data and also to navigate single messages---e.g., searching for messages sent in a specific time interval. In the first scenario it is perferable to model chats as documents, while in the second to atomically model messages as documents. This requirement enables this kind of flexibility, also allowing to interlink items, e.g., for visualizing the entire chat containing a message. Finally, different document collections should be allowed to refer to different \aclp{kr}.
    \item Support for \emph{annotation versioning}. Our previous architectural model~\cite{BELLANDI2024} was already able to manage multiple versions of the same document. This functionality was introduced for storing the original document and additional pre-processed copies. In this formalization of the \ac{dia} we simplify this functionality to versioning only the annotations---which can be modified by humans through the Error correction \ac{ui}---in order to allow \emph{error correction capability} and \emph{traceability} at the same time. For fully supporting these principles, we borrow consolidated concepts from version control systems for software development~\cite{version_control_survey_2018}: it should be possible to maintain, and potentially restore, previous versions of all the annotations and any modification should be recorded with an identifier of the user who performed the operation and an explanatory message describing the changes---similarly to a git commit~\cite{version_control_survey_2018}. For more complex needs requiring documents' modifications, users can rely on the support for \emph{multiple collections} and ingest the processed documents in a new collection.
    \item \textit{Retrieval \acp{api}}, supporting both sparse \ac{ir}---for standard keyword-based retrieval or faceted search (i and iii)~\cite{ARMENTANO2014_nlp_faceted,hearst_2006_faceted}---dense semantic retrieval~\cite{negar_sparse_dense_retrieval_2021}---for question answering (iv)---and hybrid retrieval combining both sparce and dense~\cite{negar_sparse_dense_retrieval_2021}. In fact, users may benefit from both: sparse retrieval systems allow to exactly seach for a certain keyword; while \ac{rag} applications generally use dense retrieval~\cite{old_ir_methods_rag_2024}; and combining sparse may provide complementary information~\cite{negar_sparse_dense_retrieval_2021}.
    \item Support for a \emph{\acf{ql}} capable of expressing complex queries to enable advanced statistical analysis (v).
    \item \emph{Visualization \acp{ui}} that allow users to visualize documents with automatic annotations, to trace and verify the mentioned entities, in support of the requirement for \emph{traceability and verifiability}.
    \item \emph{Error correction \acp{ui}} that allow users to access documents and automatic annotations, to correct machine errors in support of the requirement for \emph{error correction capability}.
\end{enumerate}
Moreover, to enable users in pursuing their \acl{ia} needs for the considered use cases multiple \aclp{ui} have been prototyped on top of these functional requirements. These are described in \Cref{sec:iaui}

As anticipated, to support the integration of heterogeneous data sources within a unified representation, we formalize three core concepts: \emph{documents}, \emph{annotations}, and \emph{entities}. These concepts establish the structural basis for storing and integrating heterogeneous data from knowledge-intensive domains. Their design is inspired by the GateNLP~\cite{cunningham-etal-2002-gate,gatenlp} model, as described in its documentation, from which we adopt the notions of documents\footnote{\url{https://gatenlp.github.io/python-gatenlp/pythondoc/gatenlp/document.html}}, annotations\footnote{\url{https://gatenlp.github.io/python-gatenlp/pythondoc/gatenlp/annotation.html}}, and annotation sets\footnote{\url{https://gatenlp.github.io/python-gatenlp/pythondoc/gatenlp/annotation_set.html}}---that allows to organize annotations in sets---as well as the use of key--value metadata, corresponding to the ``features'' property of GateNLP objects.
%
GateNLP~\cite{cunningham-etal-2002-gate,gatenlp} is a Python framework for natural language processing that provides a flexible representation of documents and annotations, also offering interactive visualization in Jupyter notebooks.

We also chose GateNLP python objects serialized in JSON format~\cite{rfc8259} as the \emph{data interchange format}, to achieve interoperability of services. The integration of dedicated services, indeed, is supported by the designed \ac{dia} for the purpose of generalizability, as dedicated processors may be required by specific use case.
For example, the input of the \emph{create document} \ac{api} or the output of systems that produce annotations, such as for \ac{ner} and \ac{nel}, must be in \emph{GateJSON} format---which means an instance of the GateNLP document object\footnote{\url{https://gatenlp.github.io/python-gatenlp/pythondoc/gatenlp/document.html}} serialized in JSON. The serialized documents, indeed, also contain the annotation sets and the annotations.
This choice also maintain compatibility with GateNLP functions---after deserializing documents into Python objects---which support advanced operations on annotation sets, useful, for instance, to identify overlapping annotations, or annotations contained in another annotation. 
As a real example, suppose to have a document, derived from a legal judgment, with two annotation sets, one containing annotations for the sections of the judgment, and the second containing \ac{ner} annotation for the people involved in the judgment. For identifying plaintiffs and defendants we usually focus on the beginning of the document with the section ``Identification of the parties''. So given this section, with GatenNLP \acp{api} we are able to obtain which people are mentioned in that section:
\begin{lstlisting}[language=Python, caption={Example of using GateNLP~\cite{cunningham-etal-2002-gate,gatenlp} \acp{api}.}, label={lst:gatenlp-section-example}]
from gatenlp import Document
import json

# Load the JSON serialized document
with open("judgment_doc.json") as file:
    raw_doc = json.load(file)

# Deserialize the document
doc = Document.from_dict(raw_doc)

# Get the annotation sets
sections = doc.annset("sections")
people = doc.annset("persons")

# Select the section of interest, in this case the first
parties_id_section = sections[0]

# Retrieve all person annotations contained in that section
mentions = people.within(parties_id_section)
\end{lstlisting}

\begin{enumerate}[label=\textbf{\arabic*.}, leftmargin=2em]
  \item \textbf{Document.}  
  A document represents a general unit of data and may vary in granularity depending on the use case---for instance, a legal judgment, a contract, or a single instant messaging message.  
  Each document is characterized by:
  \begin{itemize}
    \item a unique \ac{uri}, ensuring global identifiability;
    \item a \emph{name}, used as a human-readable label;
    \item the \emph{text content};
    \item optional \emph{metadata}, stored as key--value pairs in a feature dictionary.  
  \end{itemize}
  Documents serve as the source layer for annotations and the reference point for traceability.

  \item \textbf{Annotation.}  
  An annotation connects a portion of a document to a specific concept or entity, thereby linking unstructured text with structured knowledge. An annotation is defined by:\todo{add uri of annotation set?}
  \begin{itemize}
    \item a unique \ac{uri};
    \item the \ac{uri} of the \emph{source document}, ensuring that the annotation can always be traced back to its origin---even when processed within collections derived from another collection;
    \item the \ac{uri} of the \emph{annotation set} of which the annotation is part, simlarly for ensuring traceability;
    \item the \emph{start} and \emph{end} character offsets in the document text;
    \item the \emph{type} of the annotation, e.g., the \ac{ner} type, or the name of a section;
    \item optional \emph{metadata}, as key--value pairs, representing additional properties, e.g., the linked entity \ac{uri} for \ac{nel} annotation or the version of the annotation algorithm.  
  \end{itemize}
  Annotations allow users to visualize, verify, and trace where entities are mentioned in documents. Beyond entity linking, this concept generalizes to any document fragment, from short spans of text to entire sections, enabling flexible annotation schemas.

  \item \textbf{Annotation Set.}

This structure allows for a better organization of the annotations and for the coexistence of multiple annotation layers over the same document (e.g., manual vs.\ automatic annotations, or distinct semantic layers), as well as for annotation versioning---which can be achieved by creating a modified annotation set, saving the vesion number, the user who modified it, and a message explaining the changes in the annotation set's metadata.

    \item a \emph{name}, describing the set (e.g., ``\ac{ner}-v1'' or ``\ac{nel}-v1'');
    \item a list of contained \emph{annotations};
    \item optional \emph{met\begin{itemize}

    \item annotations
    \item annotation sets for organizing different annotation types sections nel
\end{itemize}

  \end{itemize}

  \item \textbf{Entity.}  
    Each entity, in line with the definition in \Cref{back:entities} is characterized by:
    \begin{itemize}
        \item a unique \ac{uri}, ensuring global identifiability;
        \item a \emph{name}, used as a human-readable label;
        \item optional \emph{metadata}, stored as key--value pairs in a feature dictionary. For example, metadata may contain the entity type (or types).
    \end{itemize}
    % Entities provide a generalizable way to interlink heterogeneous documents and annotations.

  \item \textbf{Document Collection.}
  A document collection contains multiple documents and provides a way to organize and manage different documents separately in the same \ac{dia} instance.
  Each document collection is characterized by:
  \begin{itemize}
    \item a unique \ac{uri};
    \item the \ac{uri} of the \emph{knowledge repository} considered by this collection;
    \item a \emph{name}, used as a human-readable label;
    \item a list of contained \emph{documents};
    \item optional \emph{metadata}, stored as key--value pairs.
  \end{itemize}

  \item \textbf{Knowledge Repository.}
  We additionally include the \ac{kr} in the data model to enable using different \acp{kr} with different document collections.
  While this view is not common for \acp{kr} and \acp{kb}, it is similar to \acp{dbms} holding multiple databases.
  A \ac{kr} is characterized by:
  \begin{itemize}
    \item a unique \ac{uri};
    \item a \emph{name}, used as a human-readable label;
    \item a list of contained \emph{entities};
    \item optional \emph{metadata}, stored as key--value pairs.
  \end{itemize}

\end{enumerate}

The metadata, present in each model, derive from GateNLP's general-purpose \emph{features} property, implemented as a key--value dictionary. While GateNLP uses internal numeric identifiers for its objects, we extend it by introducing globally unique \acp{uri}, simplify locating resources, enabling inter-document linking, and simplifies the compatibility with distributed storage.
Furthermore, \acp{uri} allow for the seamless integration of public \acp{kr} and \acp{kb}, allowing to link annotations to entities from Wikipedia, Wikidata, or even domain specific \acp{kr}.

In \Cref{fig:data-model}, entities are placed in the \acl{kr}. This is because entities are stored and managed in the \acl{kr}, as we will describe in the next section.




\section{Architectural Model}

\begin{figure}[t]
  \centering
  \includegraphics[width=\textwidth]{images/architectural_model.drawio.pdf}
  \caption{High-level architectural model of the \acl{dia}.}
  \label{fig:architectural-model}
\end{figure}

An overview of the \acl{dia} is depicted in \Cref{fig:architectural-model}. The figure shows a clear division between what is part of the \ac{dia} and what is outside it.
Inside we find the \emph{storage layer}, whose objective is to store information regarding documents, annotations, and entities, as well as to organize this information for efficient access via indexing.
This layer is composed of:
\begin{itemize}
  \item \emph{\Acf{ad}}, which stores documents, annotations, and annotation sets as defined in the data model.
  \item \emph{\Acf{kr}}, which stores local entity knowledge, according to the definitions of entity and \ac{kr} from \Cref{back:entities}. Note that, while in the \ac{dia} design this is not enforced, the \ac{kr} component can be a \acl{kb} holding facts about entities in the form of \ac{rdf} triples.
  \item \emph{\Acf{ri}}, which holds indexes for efficient retrieval and supports sparse, dense, and hybrid retrieval paradigms.
\end{itemize}

On top of the storage layer sits the \ac{api} layer, which exposes interfaces to interact with the persisted data. This is the component that must provide the mandatory \acp{api} defined by the requirements:
\begin{itemize}
  \item \emph{\ac{crud} \acp{api}} for documents, annotations, and annotation sets---stored in the \ac{ad}---and for entities---stored in the \ac{kr}.
  \item \emph{Sparse, dense, and hybrid retrieval \acp{api}} using the indexes from the \ac{ri} for efficient retrieval.
  \item \emph{\acl{ql} \acp{api}} supporting aggregation and filtering operations required to compute statistics, derive distributions, and answer structured analytical queries over documents, annotations, and entities. These interfaces must support at least the standard family of operations found in declarative query languages---such as grouping and aggregation (e.g., counts, distinct counts, maxima/minima, or averages)---as documented in mainstream \acp{dbms} (e.g., PostgreSQL~\cite{postgresql-agg-doc}). Such capabilities are essential to support use case~(v), i.e., statistical analysis and monitoring.
\end{itemize}
These \acp{api} must operate over the network to support distributed deployments, enabling parallel processing and horizontal scalability. Furthermore, they must support the GateJSON format for both input and output of annotated documents, ensuring interoperability across heterogeneous components.

Besides the storage and \ac{api} layers, the \ac{dia} features the two fundamental \acp{ui} for fulfilling traceability-verifiability and error-correction capability.
The \emph{Visualization \ac{ui}} must provide the following functionalities to users:
\begin{itemize}
  \item visualize annotated documents, selecting which annotation sets to visualize;
  \item access the properties (defined in the data model) of documents, annotation sets, annotations, and entities, while visualizing the annotated document---e.g., by clicking on an annotation to inspect annotation properties---or given the \ac{uri} of the resource;
  \item given an entity, trace which annotations mentioned it and visualize them in the document.
\end{itemize}

On the other hand, the \emph{Error correction \ac{ui}} provides functionalities to correct annotation errors from automatic tools and, in general, provides delete operations for removing automatically created resources whose creation was an error---such as duplicated entities in the \ac{kr}. While additional functinalities, such as manual ingesting of documents, may be useful, we only include the minimum necessary design principles for allowing \emph{error correction capability}.
\begin{itemize}

  \item For documents it only provides the delete operation, which consequently deletes all the associated annotaiton sets and annotations.
  \item Also for document collections it provides the delete operation, which consequently deletes all documents and annotations.
  \item Similarly for knowledge repository it allows for deletion, which propagates to all contained entities.
  \item For annotations:
  \begin{itemize}
    \item create;
    \item delete;
    \item move---same as modify start and end;
    \item modify properties (type and metadata)---for example to changed the linked entity;
    \item for versioning, each modification---with annexed the user id and a message explaining the changes---must be persisted in a \emph{history} at the annotation set level. This centralizes the annotation versioning management to the annotation set, allowing to visualize the history of the entire annotation set and to perform rollback operations. 
  \end{itemize}
  \item For annotation sets:
  \begin{itemize}
    \item delete together with all the annotations;
    \item view history, including user ids and message for each modification;
    \item rollback to a specific version, undoing all the modifications performed afterwards.
  \end{itemize}
  \item For entities it allows to:
  \begin{itemize}
    \item create a new entity to manually assign it to annotations;
    \item merge two entities and consequently assign all the linked annotations to the new entity;
    \item delete an entity and optionally chose another entity to assign to the linked annotations and get the list of affected annotations for manually revising them;
    \item modify name and metadata.
  \end{itemize}
\end{itemize}


Finally, we conclude the core architectural section by defining three information flows that the \ac{dia} must support. These flows cover all possible interactions with external services---such as preprocessing components, text annotators, or \acp{ui}---and assume interoperability through the GateJSON format. We also recall the design guideline of ensuring loose coupling among services to reduce cascading failures.

\begin{enumerate}
\item \emph{\Acl{ie} flow.}
Raw data are processed by external services, serialized in GateJSON, and injected into the \ac{dia} via \acp{api}. This corresponds to the green arrow in \Cref{fig:architectural-model}, flowing from outside the system into the core of the \ac{dia}.

\item \emph{\Acl{ia} flow.}
\acp{ui} retrieve stored information in GateJSON format via read or retrieval \acp{api} and present it to end users. This corresponds to the red arrow in \Cref{fig:architectural-model}.

\item \emph{Circular flow.}
Some services both read from and write to the \ac{dia}. This includes, for example, processing systems arranged in a pipeline: they fetch a document, enrich it (e.g., by adding an annotation set), and persist the additional information. This bidirectional flow is represented by the two small arrows between the core and the \emph{Processing Services}. The \emph{Error-correction \ac{ui}} is an analogous case: it reads from the \ac{dia} and persists human-made revisions back into it.
\end{enumerate}

Together, these three flows ensure that external services can ingest, access, and iteratively refine information, making it possible to generalize the proposed \ac{dia} to different use cases. In addition, the ability to run processing on top of previously generated results, combined with the network-based \ac{api} layer, enables parallel and distributed execution. This is facilitated by operating on separate annotation sets, which minimizes conflicts between concurrent processes. These properties support the scalability principle, especially when the \ac{dia} is combined with an orchestrator that schedules services over different documents. Finally, the use of \acp{uri} for all resources allows documents and annotations to be retrieved unambiguously via the \acp{api}.
